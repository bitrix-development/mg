# Обзор проекта LK (mg) по README и SQL-дампу

## 1. Описываю ситуацию по тому, что видно из README

### 1.1. Общая картина проекта

- Проект LK — личный кабинет для руководителя и сотрудников отдела продаж.
- Есть базовая доменная модель:
  - `users` с ролями `manager`, `admin`, `sales`, плюс поля для `reset_token` и `reset_token_expires`.
  - `clients`, `orders`, `order_items`, `products` — классическая CRM-like схема для клиентов, заказов и товарных позиций.
  - Таблицы для аудита/безопасности: `login_attempts`, `user_logs`, `rate_limit_sessions`.
- Акцент явно сделан на безопасности:
  - Защита от CSRF и XSS заявлена.
  - Есть концепция ограничения числа попыток входа (RateLimiter).
  - Планируется/реализуется логирование действий пользователей.
  - Планируется вынос чувствительных данных в `.env`.

### 1.2. Структура файлов (по README)

**Корень проекта:**

- `db.php` — подключение к БД (в планах перевод на `.env`).
- `config.php` — параметры (обозначен как устаревающий, должен уйти в `.env`).
- `.env` — переменные окружения.
- `functions.php` — вспомогательные функции (логирование, работа с лимитом и т.п.).
- `includes/security.php` — проверки авторизации, CSRF-защита.
- `includes/Logger.php` — логирование действий и попыток входа (упомянуто, что есть на диске).
- `includes/RateLimiter.php` — класс для ограничения частоты входа.
- `admin_change_password.php` — смена пароля админом.
- `phpinfo.php` — информация о PHP.

**Каталог `pages/`:**

- `dashboard.php` — дашборд-обзор.
- `clients.php` — список клиентов.
- `client_detail.php` — детали клиента и заказы.
- `orders.php` — список заказов, с фильтрами по статусу.
- `order_detail.php` — детали заказа (заявлено как «будет»).
- `profile.php` — профиль пользователя, смена пароля.
- `login.php` — форма входа (в планах интеграция rate limit, логирования и CSRF).
- `logout.php` — выход из системы (указано, что «работает правильно»).
- `activity_log.php` — архив действий (пока «будет»).

**Каталог `includes/`:**

- `header.php`, `footer.php` — оформление (layout).

### 1.3. Что уже реализовано (по README)

- Авторизация по логину и паролю.
- Ролева́я модель пользователей (`manager`, `admin`, `sales`).
- Список клиентов с фильтрацией.
- Список заказов (фильтры реализованы частично).
- Защита от CSRF и XSS (по заявлению).
- Сброс пароля вручную администратором.
- Раздел профиля и смена пароля.
- Архитектура базы данных «полностью готова для бизнеса».
- Корректная работа выхода (`logout.php`).
- Классы для логирования и ограничения частоты входа уже существуют (но их внедрение ещё впереди).

### 1.4. План работ (по README)

1. Внедрение использования `.env` в `db.php` (скрытие паролей БД).
2. Перевод `config.php` на `.env` (убрать личные данные из кода).
3. Интеграция `RateLimiter` и логирования в `login.php` и ключевые действия.
4. Реализация `activity_log.php` — страница просмотра истории действий.
5. Миграция старого кода на новые функции логирования и rate limit.
6. Добавление сброса пароля по email (через `reset_token` и соответствующие формы).
7. Создание полноценной страницы `order_detail.php`.
8. Тестирование всех фильтров заказов, поиска и прав доступа.
9. Добавление админ-архива логов (очистка старых записей).

### 1.5. Вывод по README

  - какие сущности есть в БД,

## 4. Что есть и выглядит хорошо (по дампу БД)

### 4.1. Соответствие README и реальной схемы

По SQL-дампу `dbcabinet`:

- Таблицы в целом соответствуют README:
  - `users`, `clients`, `orders`, `order_items`, `products`, `rate_limit_sessions`.
  - Вместо `user_logs` — `user_actions` (аналог по сути).
- Все основные связи реализованы через внешние ключи.

### 4.2. Связи и ограничения

- Внешние ключи:
  - `orders.client_id` → `clients.id`.
  - `orders.user_id` → `users.id`.
  - `order_items.order_id` → `orders.id`.
  - `user_actions.user_id` → `users.id`.
- Для всех таблиц настроены `PRIMARY KEY` и `AUTO_INCREMENT`.

### 4.3. Индексы

- `users`:
  - PK по `id`,
  - UNIQUE по `login`,
  - индекс по `reset_token` (`idx_reset_token`).
- `orders`:
  - PK по `id`,
  - индексы по `client_id`, `user_id`.
- `order_items`:
  - PK по `id`,
  - индекс по `order_id`.
- `rate_limit_sessions`:
  - PK по `id`,
  - уникальный индекс по `identifier`,
  - индексы `idx_login`, `idx_ip`, `idx_is_blocked`.
- `clients`, `products`, `user_actions` имеют PK и индексы по связям.

### 4.4. Безопасность и хранение паролей

- В таблице `users`:
  - поля `login`, `password`, `role`, `reset_token`, `reset_token_expires`;
  - пароли выглядят как bcrypt-хеши (`$2y$10$...`), что соответствует современному подходу (`password_hash` в PHP).
- Наличие индекса по `reset_token` говорит о готовности к реализации механизма сброса пароля.

### 4.5. Rate limiting и логирование

- `rate_limit_sessions`:
  - хранит `identifier`, `login`, `ip_address`, `attempt_count`, `first_attempt`, `last_attempt`, `is_blocked`, `blocked_until`;
  - индексирован по `login`, `ip_address`, `is_blocked`.
- `user_actions`:
  - логирует действия с полями:
    - `user_id` (nullable),
    - `action`,
    - `description`,
    - `ip_address`,
    - `created_at`.
  - В дампе уже есть реальные записи типа:
    - `login_failed`, `login_success` для пользователя `admin1`.
- Это подтверждает, что логирование и rate limiting не только запланированы, но и частично внедрены.

### 4.6. Демо-данные

- `clients`, `products`, `orders`, `order_items` заполнены демонстрационными данными:
  - этого хватает для тестирования:
    - списков клиентов и заказов,
    - базовых отчётов и фильтров,
    - отображения информации на дашборде.

## 5. Чего не хватает / зоны для доработки в схеме БД

### 5.1. Связь `order_items` ↔ `products`

- В `order_items` поле `product_name` есть, но во всех строках установлено в `NULL`.
- Отдельно существует таблица `products` с идентификаторами, названиями, размерами и ценами.
- В текущей схеме:
  - нет FK из `order_items` на `products`;
  - не реализована прямая связь между позициями заказа и справочником товаров.
- Возможные варианты доработки:
  - добавить в `order_items` поле `product_id` (FK на `products.id`);
  - либо зафиксировать архитектурно, что `order_items.product_name` — это текстовый слепок товара на момент продажи и связь с `products` не требуется (но тогда нужно заполнить `product_name` и продумать, как использовать `products`).

### 5.2. Привязка клиентов к менеджерам

- В `clients` нет поля, которое бы явно указывало на ответственного менеджера (`manager_id`/`owner_user_id`).
- Сейчас связь «менеджер ↔ клиент» может лишь косвенно выводиться через `orders.user_id`, но:
  - это неудобно для фильтрации по «моим клиентам»;
  - не учитывает ситуацию смены ответственного.
- Возможные варианты:
  - добавить в `clients` поле `manager_id` (FK на `users.id`);
  - при необходимости M:N-связи завести таблицу `client_user`:
    - `client_id`, `user_id`, роль в отношении (основной/поддержка и т.п.).

### 5.3. Отсутствие индексов по ключевым полям для фильтрации

- Ожидается активная фильтрация:
  - по заказам: `status`, `order_date`;
  - по клиентам: `status`, `email`, `phone`.
- В текущей схеме:
  - у `orders` нет индексов по `status` и `order_date`;
  - у `clients` нет индексов по `status`, `email`, `phone`.
- Для продакшена рекомендуется:
  - добавить индексы:
    - `ALTER TABLE orders ADD INDEX idx_status (status);`
    - `ALTER TABLE orders ADD INDEX idx_order_date (order_date);`
    - `ALTER TABLE clients ADD INDEX idx_status (status);`
    - `ALTER TABLE clients ADD INDEX idx_email (email);`
    - `ALTER TABLE clients ADD INDEX idx_phone (phone);`
  - или составные индексы, если есть типичные комбинации фильтров (`status + дата`, `статус + менеджер` и т.п.).

### 5.4. Статусы как ENUM и возможная эволюция

- Поля `status` в `clients` и `orders` реализованы как ENUM:
  - `clients.status` → `('active','inactive')`;
  - `orders.status` → `('new','processing','completed','cancelled')`.
- Для текущего объёма это приемлемо, но:
  - при частых изменениях набора статусов (особенно при необходимости локализации, сортировки этапов, хранения цветовых схем и т.п.) лучше иметь таблицы-справочники:
    - `order_statuses` и `client_statuses` с отдельными полями `code`, `title`, `sort_order`.
- Пока это не критично, но стоит понимать риски будущих изменений ENUM (ALTER TABLE по большому объёму данных).

### 5.5. Отсутствие soft-delete / признаков архивности

- В таблицах `clients`, `orders`, `products` нет полей вроде:
  - `deleted_at`,
  - `is_deleted`,
  - `archived_at`.
- Если бизнес-процесс требует «удаления» сущностей, но при этом:
  - нужны исторические отчёты;
  - требуется сохранять ссылки на старые заказы/клиентов;
  - то следует либо:
    - ввести soft-delete (поле `deleted_at` и фильтрация в коде),
    - либо строго договариваться, что `DELETE` по важным сущностям не используется и заменяется переводом в определённый статус.

### 5.6. Работа с временными зонами

- Везде используются `timestamp` с дефолтом `CURRENT_TIMESTAMP`.
- Это удобно, но:
  - нужно чётко определиться, в какой зоне хранятся данные (обычно UTC);
  - конвертацию в локальное время лучше делать на уровне приложения.
- Это не проблема схемы, но важная договорённость для фронтенда/отчётности.

## 7. Какие вопросы остаются уже после просмотра дампа

### 7.1. Детальная связь `order_items` и `products`

- Сейчас `product_name` в `order_items` всегда `NULL`, а таблица `products` заполнена.
- Вопросы:
  - При создании заказа позиции выбираются из `products` или вводятся произвольно?
  - Нужна ли отчётность по конкретным товарам (топ-продажи, маржинальность по позициям)?
  - Если да — логичнее добавить FK `product_id` в `order_items` и использовать `products` как основной справочник.

### 7.2. Модель владения клиентами и доступов

- По схеме БД по-прежнему не видно:
  - есть ли «ответственный» менеджер за конкретного клиента;
  - как будут фильтроваться клиенты и заказы для разных пользователей.
- От этого зависят:
  - поведение страниц `clients.php`, `client_detail.php`, `orders.php`, `dashboard.php`;
  - тонкая настройка прав (кто может видеть/редактировать что).

### 7.3. Детализация ролей и управление правами

- Роли по-прежнему хранятся в ENUM без отдельного справочника.
- Вопросы:
  - будет ли достаточно простых проверок по ENUM в коде;
  - планируется ли когда-нибудь перейти к более гибкой модели прав (ACL, RBAC с таблицами `roles`, `permissions`, `role_permission`, `user_role`).

### 7.4. Конкретные политики по сбросу пароля

- В схеме есть всё необходимое:
  - `reset_token`, `reset_token_expires`, индекс по токену.
- Открытые вопросы:
  - какова политика TTL (например, 15 минут, 1 час, 24 часа);
  - будет ли токен храниться в явном виде или в виде хеша (для повышения безопасности в случае утечки БД).

### 7.5. Политики по хранению и очистке логов

- `user_actions` и `rate_limit_sessions` потенциально могут сильно расти.
- Вопросы:
  - сколько времени планируется хранить логи и данные rate limiting;
  - будет ли реализован механизм архивации/удаления по сроку давности;
  - как именно будет выглядеть админский интерфейс по логам (фильтрация, экспорт, массовая очистка).

### 7.6. Точные требования к отчётности/дашборду

- С точки зрения БД уже можно:
  - считать выручку по заказам;
  - строить разбивку по менеджерам (`orders.user_id`);
  - смотреть историю продаж по клиентам.
- Тем не менее, остаётся открытым:
  - какие конкретно графики/таблицы/метрики должны быть на `dashboard.php`;
  - нужны ли отчёты по продуктам (что усилит потребность в связи `order_items` → `products`).
