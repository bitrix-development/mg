## 1. Описываю ситуацию по тому, что видно из README

### 1.1. Общая картина проекта

- Проект LK — личный кабинет для руководителя и сотрудников отдела продаж.
- Есть базовая доменная модель:
  - `users` с ролями `manager`, `admin`, `sales`, плюс поля для `reset_token` и `reset_token_expires`.
  - `clients`, `orders`, `order_items`, `products` — CRM-like схема для клиентов, заказов и товарных позиций.
  - Таблицы для аудита/безопасности: `user_actions`, `rate_limit_sessions` (в README/старых документах ещё встречаются устаревшие названия таблиц).
- Акцент явно сделан на безопасности:
  - Защита от CSRF и XSS заявлена.
  - Есть концепция ограничения числа попыток входа (RateLimiter).
  - Есть логирование действий пользователей (частично).
  - Чувствительные данные вынесены в `.env` (файл не хранится в репозитории по правилам безопасности).

### 1.2. Структура файлов (по README)

**Корень проекта:**

- `db.php` — подключение к БД (уже использует `.env`).
- `config.php` — **устарело, удалено** (в репозитории отсутствует, использование не найдено).
- `.env` — переменные окружения (**не коммитится**).
- `includes/security.php` — проверки авторизации, CSRF-защита.
- `includes/Logger.php` — логирование попыток входа (на текущий момент — в файл).
- `includes/RateLimiter.php` — класс для ограничения частоты входа.
- `forgot_password.php`, `reset_password.php` — сброс пароля по email реализован (см. ниже, п. 1.4/3.4).

**Каталог `pages/`:**

- `pages/dashboard.php` — дашборд-обзор (уточнено: дашборд лежит именно здесь).
- `pages/login.php` — форма входа.

### 1.3. Что уже реализовано (по README)

## 3. Что вызывает вопросы и хотелось бы уточнить (по README) — актуализировано

### 3.1. Связь пользователей с клиентами и заказами — уточнено и исправлено

**Было в ревью:**  
- В `orders` есть `user_id` (ведущий менеджер заказа).  
- В `clients` не видно ответственного менеджера.

**Уточнено (бизнес-правило):**  
- За клиентом **всегда фиксируется ответственный** менеджер — это пользователь `users.id`.

**Сделано (БД):**
- Используется поле `clients.user_id` как ответственный.
- Добавлены индекс и внешний ключ `clients.user_id -> users.id` (ранее поле могло быть «просто цифрой» без FK).
- Проверены «битые ссылки» (значения `clients.user_id`, отсутствующие в `users.id`) — проблем не обнаружено на момент изменения.

**Итог:**  
- Владелец/ответственный клиента определяется однозначно: `clients.user_id`.

### 3.2. Роль и различие таблиц логов — текущая фактическая картина (требует приведения к единой модели)

**Уточнение по факту кода и дампа:**
- В документации/ревью встречается `user_logs`, но в БД фактически используется `user_actions`.
- В репозитории есть `activity_log.php`, который обращается к таблице `user_logs` (несоответствие реальной схеме БД).
- В БД (дамп) присутствует `user_actions` (и в ней есть записи `login_failed/login_success`).
- В коде присутствует `includes/Logger.php`, но он пишет **в файл**, а не в таблицу `user_actions`.
- `rate_limit_sessions` используется `includes/RateLimiter.php` как **агрегированное** хранилище для блокировок/лимитов по логину и IP.

**Вывод (на сейчас):**
- Логирование и аудит разделены на 2 параллельных потока:
  1) файловый лог (`/logs/user_actions.log`) — используется админ-страницей `admin/logs.php`;
  2) таблица `user_actions` — есть в БД и подходит для UI-страницы логов, но код должен быть приведён к ней.

**Рекомендация (как задача на дальнейший шаг):**
- Выбрать единый источник истины (предпочтительно `user_actions` в БД) и:
  - исправить `activity_log.php` на чтение `user_actions` вместо `user_logs`;
  - обновить `Logger` (или добавить DB-логгер) так, чтобы события писались в `user_actions`, либо чётко описать, зачем нужен файл.

### 3.4. Сброс пароля — уточнено по факту внедрения

- Сброс пароля реализован через `reset_token` + `reset_token_expires`.
- TTL токена: 1 час (по текущей реализации).
- Токен хранится в явном виде (как значение в БД), используется однократно (после смены пароля сбрасывается в NULL).

**Техническая актуализация (сделано):**
- `forgot_password.php` и `reset_password.php` приведены к единому подключению через `db.php` (для исключения проблем с путями к `.env` и дублирования PDO).
- Исправлены редиректы для авторизованного пользователя на `/pages/dashboard.php`.

### 3.5. Политики rate limiting — уточнено по факту кода

По `includes/RateLimiter.php`:
- лимит: 5 попыток
- интервал: 300 секунд (5 минут)
- блокировка: 900 секунд (15 минут)
- ключ: комбинация `identifier/login` + `ip_address` (проверка в `rate_limit_sessions`)

**Примечание:** на текущий момент интеграция RateLimiter в `pages/login.php` помечена комментариями как плановая; требуется проверить/внедрить фактические вызовы `allowLogin()`.
